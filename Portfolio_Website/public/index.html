Projects

Automated Testing Harness

As I was practicing DS&A problems from CTCI, I realized I was spending more time on testing my solutions, than writing them. I decided to automate this process to save time on future problems, and gain a useful industry practice.


Right now, the harness supports a stress test on int parameter functions, to check for runtime errors.


It takes in the input java file and an output file name, parses the input file for int parameter functions, and creates an output .java file that stress tests each function that contains solely integer parameters.


For the next feature, I am working on getting the harness to handle IO mapping.


(Java, Perl, Linux) Code: Github

Nutrition Basket Calculator

Since I'm a cooking nut, I like to cook a lot. But I don't know how healthy my diet is. I wanted to measure how healthy I'm eating, so I decided to make an app to do just that.


My nutrition basket app takes in groceries input for a 5-10 day period, and calculates my average daily nutritional intake.


Data is scraped in real-time from Wikipedia's nutritional profile for each ingredient, and multiplied by the amount of the ingredient in the basket.


Contains an additional "view" to see what percent of USDA guidelines, we are meeting for each macro and micro nutrient.


(Java, Node.js, PostgreSQL) Code: Github

Delivery Path Visualizer

I wanted to practice implementing a graphing algorithm, since we never got to do that in our DS&A classes. I chose Djikstra, and thought of real-life use cases for the minimum path algorithm.


It seemed natural to me, since I order grocery deliveries, that I can apply Djikstra to find the optimum path to go from the store to a single location. Furthermore, in a higher-level algorithm, I can estimate an optimal solution to traverse through several waypoint destinations in between.


Visualized UI through OpenGL library.


(C++, Perl, OpenGL) Code: Github

Huffman Encoding

I recently implemented huffman encoding in SmallTalk, for a comparative programming class I took. I wanted to see if I could implement this concept in a more popular language, and to stretch the bounds of how compression can work.


I created a script to apply huffman compression/decompression, working differently based on different input options.


Options: -w : Alphabetize by high-frequency sequences of characters (words),

-c : by characters,

-r [start] [end] : Compress/decompress within an internal range of the text file.


(C, Perl) Code: Github

Development Philosophies

Safety First:

Protected code can offer stability and options for large software apps, preventing a single bug from crashing the whole program. As a best practice, I try-catch all my code in main, and often include additional try-catch's in lower levels, for error-logging, and to skip buggy test cases.


Diagrams and Pseudocode:

I am not very good at seeing big picture sometimes, or in translating novel behaviors and specs into code. So I've become comfortable going through iterations of drawing diagrams and writing pseudocode. This was a process I did often when I had to simulate the RISC-V ISA in C, going through quite a bit of paper.


Trust but Verify:

Testing saves lives. I test all my code that involves data processing, or some kind of I/O mapping.


Design-Plan-Allocate, Execute:

This is my framework for solving a problem. Is it easy enough for me to execute right away? Otherwise, I need to bust out paper and think. Is it complex enough to require a plan for multiple coding sessions? Can I set firm expectations for deadlines, or do I need to allocate buffer time or rolling deadlines to deal with unforeseen variables?
